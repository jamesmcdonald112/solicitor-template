## The Preloader (Every Site Starts With This)

Every solicitor site built from this template opens with a full-screen preloader.
This is a non-negotiable part of the template.

```
src/components/preloader/
  Preloader.astro
  preloader.css
  preloader.gsap.ts
```

The preloader:
- Covers the full viewport on page load
- Shows the firm logo (centred) with a GSAP entrance animation
- Has a brief moment of stillness
- Animates out to reveal the site — the exit should feel like a curtain lifting
- Total duration: 1.8–2.5 seconds maximum
- The site beneath should be hidden until the preloader exits

The exact animation is Claude's creative decision — it should feel premium and considered,
not a generic spinner. Think: logo draws in, a horizontal line sweeps across, panels split.
Something that sets the tone for what follows.

## The Token System

The design system is already built. Read it before designing.

```
src/styles/tokens.css        ← all semantic tokens
src/styles/themes/           ← primitive tokens (colours, font stacks, sizes)
```

The tokens are the vocabulary. Use them. Do not use raw values.
The token system gives Claude a pre-designed palette and scale to work within —
but the creative decisions about *how* those tokens are composed are Claude's to make.

After studying the references and reading the tokens, Claude should ask itself:
"What is the most beautiful thing I can build within these constraints?"

## GSAP Motion Principles

These are technical constraints, not aesthetic ones. They apply to every component.

```ts
// Import — always from this path
import { gsap, ScrollTrigger } from '../../scripts/gsap.ts';

// Reduced motion guard — always first
if (window.matchMedia('(prefers-reduced-motion: reduce)').matches) return;

// Scroll entrance — elements arrive from below, unhurried
gsap.from(element, { y: 30, opacity: 0, duration: 0.9, ease: 'power2.out' });

// Stagger — sequential reveals
gsap.from(elements, { y: 20, opacity: 0, stagger: 0.12, duration: 0.8 });

// Never: bounce, spin, rotate, scale above 1.05, slide more than 60px
// Never: multiple competing animations on the same element
// Always: unhurried. If it feels quick, it is too quick.
```

CSS — inside the .astro file
Styles live in the <style> block of the component. Astro scopes them automatically.
Do NOT create sibling .css files. All component styles go in <style>.

```astro
<style>
  .btn {
    --btn-bg: var(--color-brand);
    background: var(--btn-bg);
    padding: var(--space-sm) var(--space-lg);
  }
</style>
```

GSAP sibling pattern
Only components with animations get a sibling .gsap.ts file.
The .gsap.ts file exports an init function. The .astro file's <script> block imports and calls it.

```astro
<!-- Navbar.astro -->
<script>
  import { init } from './navbar.gsap.ts';
  init();
</script>
```

```ts
// navbar.gsap.ts
import { gsap, ScrollTrigger } from '../../scripts/gsap.ts';

export function init(): void {
  if (window.matchMedia('(prefers-reduced-motion: reduce)').matches) return;
  // animation code here
}
```

One important note: Astro deduplicates identical <script> blocks — if a component is used multiple times on a page, the script runs once. GSAP init functions must therefore use document.querySelectorAll() to target all instances, not just the first.

## Component Template

Every `.astro` file follows this structure:

```astro
---
/*
  COMPONENT: [Name]
  FILE: src/components/[folder]/[Name].astro
  PURPOSE: [One sentence]
  VARIANTS: [data-variant values if any]
  CONNECTED TO: [pages or components that use this]
  GSAP: [yes — see [name].gsap.ts | no]
  CONFIG READS FROM: [src/config/file.ts if applicable]
  DO NOT: [specific prohibitions for this component]
*/

interface Props {
  // all props typed
}

const { } = Astro.props;
---

<!-- semantic HTML -->

<style>
  /* All component styles go here. Astro scopes them automatically.
     No sibling .css files. Tokens only — no raw values. */
</style>

<!-- script only present if GSAP needed -->
<script>
  import { init } from './[component-name].gsap.ts';
  init();
</script>
```

## Component Build Order

Build by page, not by component. Each session sees a full page flow so visual coherence
is maintained within that page. Do not attempt multiple sessions in one go.

```
Session 1 — Foundation
  Preloader, Button, Section, Container, Navbar, Footer
  These are dependencies for every other session. Build them first.

Session 2 — Homepage
  Hero, TrustStrip, ServiceCard (grid), SectionHeader, TestimonialBlock, CTAPanel
  Build all homepage sections together so the full page flow is visible in one context.

Session 3 — Service Pages
  ServiceHero, ArticleBlock, ProcessSteps, FAQAccordion, ServicesList
  Build together — they share the same content rhythm and spacing logic.

Session 4 — Contact & Thank You
  ContactForm (STYLE ONLY — do not rebuild logic), ContactPanel, ThankYouPage

Session 5 — Supporting sections (if needed)
  TeamMember, TeamSection, PullQuote, ReviewsStrip
```

Each session: load the brief, read the tokens, build everything in that session's scope.
Later sessions depend on components from earlier sessions existing and being correct.
